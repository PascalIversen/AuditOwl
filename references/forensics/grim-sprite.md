
# GRIM, GRIMMER, and SPRITE (granularity + reconstruction)

Extension to the main audit prompt for checking that reported
`(mean, SD, n)` summary statistics on bounded / integer variables
are *mathematically possible* and, if possible, that the implied
sample distribution is plausible.

Findings here belong in the **main audit's §14 (statistical
integrity)**, cross-referenced from §F3.5 of the forensics skill
(metric-grid checks).

All rules of `../../audit-prompt.md` §0 apply.
Per rule I, write the checks under `scripts/` and run them.

## §G1. When this skill applies

The variable being summarised must be either:

   - **Integer-valued** (counts, Likert scale, age in years,
     baseline characteristics in a Table 1, MVP votes), OR
   - **Granular with a known step** (data reported to halves,
     tenths, etc.).

It does NOT apply to:

   - Aggregated metrics computed over different test sets per
     condition.
   - Averaged scores rounded to many decimals where the underlying
     unit is continuous (e.g., mean RMSE across many test items).

Most useful for: baseline tables in clinical/RCT papers,
behavioural-science papers reporting Likert means, count-data
ML benchmarks (e.g., "number of correctly solved tasks").

## §G2. Inputs

For every `(mean, SD?, n)` triple in the paper:

   - `mean` — exact reported value
   - `d_mean` — decimals reported for the mean
   - `sd` — exact reported value (optional)
   - `d_sd` — decimals reported for the SD
   - `n` — sample size
   - `granularity` — usually `1` for integers, else `0.5`, `0.1`, …
   - `lo, hi` — bounds of the variable (e.g., 1, 5 for a 1–5
     Likert scale). Required for SPRITE; optional for GRIM /
     GRIMMER.

## §G3. GRIM test (Brown & Heathers 2017)

Reported mean must satisfy `round(mean · n / granularity) ·
granularity / n ≈ reported_mean` to within `±0.5 · 10⁻ᵈ_mean`.

Power: with integer data reported to 2 decimals, the test
catches roughly `(100 − n) / 100` of fabricated means up to
N = 99. Above N = 100 the test loses discrimination at 2
decimals.

Algorithm:

       def grim_ok(mean, n, decimals, gran=1):
           tol = 0.5 * 10 ** (-decimals) + 1e-12
           k = round(mean * n / gran)
           candidate = k * gran / n
           return abs(round(candidate, decimals) - mean) <= tol

## §G4. GRIMMER test (Anaya 2016)

Variances of integer data follow a discrete repeating pattern
that depends on N. Given `(mean, sd, n)` reported on integer
data, the pair must:

   1. Pass GRIM on `mean`.
   2. The variance `sd ** 2` must match one of the precomputed
      patterns for that `n`.
   3. The mean's fractional part must be consistent with the
      mean-pattern for that variance.

Implementation: use Anaya's open-source reference (the
`prepubmed.org/grimmer` web app or its Python source on
GitHub). Wrap it; do not reimplement the patterns — they were
generated by exhaustive enumeration and the floating-point
edge cases are non-trivial.

When a mean passes GRIM but the SD fails GRIMMER, you have a
high-confidence arithmetic inconsistency.

## §G5. SPRITE (Heathers, Anaya, van der Zee, Brown 2018)

For each `(mean, SD, n, [lo, hi])`, attempt to find a sample
distribution of `n` integers in `[lo, hi]` whose mean and SD
match (within rounding). Three outcomes:

   (a) **No solution found** after many random starts → the
       summary statistics are jointly impossible at the stated
       bounds. High-severity finding.
   (b) **A solution is found, but only with an extreme shape**
       (e.g., bimodal with ~half at `lo` and ~half at `hi`,
       or a single extreme outlier doing all the work) → the
       paper's reported numbers technically exist but the
       implied distribution is implausible given the data-
       generating process described in the methods.
   (c) **Solutions are found and look reasonable** → no
       finding.

Procedure (Heathers et al. 2018):

   1. Initialise a random integer sample of size `n` in
      `[lo, hi]` with mean equal to `round(mean · n) / n` (so
      GRIM is satisfied).
   2. While `|sample_SD - target_SD| > tol`:
        - Pick two indices `(i, j)`; if SD must decrease, swap
          one toward the mean; else swap one away.
        - Updates leave the mean unchanged when paired.
   3. Stop when the SD matches within rounding, or after K
      cycles → mark "no solution".
   4. Repeat M times to collect multiple distinct solutions.

Use the reference `pSPRITE` implementation (open-source, BSD-
style licence). Wrap it under `scripts/`.

## §G6. Reference implementation

The reference scan lives at `scripts/check_grim.py`.

For production use, replace the heuristic `grimmer_ok` and
`sprite_attempt` with the reference implementations:

   - GRIM/GRIMMER: <https://prepubmed.org/grim_test>,
     <https://github.com/JordanAnaya/prepubmed>
   - SPRITE: pSPRITE on GitHub (also rSPRITE Shiny app)
   - Exhaustive complement: CORVIDS (Wilner et al. 2018) gives
     all possible solutions analytically; use when SPRITE fails
     to find one and you need to *prove* impossibility.

## §G7. Phrasing the finding

Good (GRIM):
   "Table 1 reports mean age `34.7` (n=21). The GRIM granularity
   is 1/21 ≈ 0.0476; nearest grid points are 0.6857 and 0.7333
   in fractional part — `34.7` is impossible at n=21. See
   `scripts/check_grim.py:grim_ok` and
   `out/grim.csv:row=2`."

Good (SPRITE):
   "Table 1 reports mean = 19.4 (SD = 19.9, n = 45) carrots taken
   per child in the control condition. With the natural upper
   bound of one carrot per snack-time observation (~40), SPRITE
   finds no distribution. Allowing up to 41 carrots, the only
   compatible distributions are extremely bimodal (most children
   take 0; a few take 40+). See `scripts/check_grim.py:
   sprite_attempt` and the histogram in `out/grim_sprite_T1.png`."

Bad:
   "The SD seems too large for the mean."

Severity: high if multiple cells in the same table fail; medium
for isolated failures. Confidence: high for GRIM/GRIMMER
(arithmetic); medium for SPRITE shape-only findings (judgement
about plausibility).
